Учебная микросервисная система по предмету «Сервисно-Ориентированное Программирование». Один сценарий – матчмейкинг в онлайн-игре – но через него показаны пять способов общения сервисов: REST, GraphQL, RabbitMQ, gRPC и WebSocket. Раздел 1 · Большая картина

Общая идея Matchmaking Arena

Представьте онлайн-игру. Игроки заходят, встают в очередь («лобби»), система собирает из них матч, матч идёт, кто-то побеждает, рейтинги пересчитываются, а в углу экрана всплывают уведомления. Matchmaking Arena воспроизводит ровно этот сценарий – но цель не игра, а демонстрация распределённой системы.

Почему матчмейкинг идеально подходит для СОП

Сервисно-ориентированное программирование – это про то, как несколько отдельных программ договариваются и работают вместе. Матчмейкинг естественно распадается на независимые задачи, каждую из которых удобно отдать своему сервису и своему способу связи:

Запомнить

Главный сервис не вызывает другие сервисы напрямую. Он лишь публикует события в RabbitMQ. Кому это интересно – те сами подписаны. Это и есть «слабая связанность» (loose coupling) – сердце сервисно-ориентированного подхода.

Как Maven Reactor собирает проект

Важно: флаги -pl и -am

Чтобы запустить один сервис, используют -pl matchmaking («project list» – только этот модуль) и -am («also make» – собери ещё и те модули, от которых он зависит, например контракты). Без -am Maven может пожаловаться, что не нашёл свежий контракт.

Принцип

Сначала договариваются о контракте (какие данные и операции есть), потом пишут реализацию. «Сначала контракт» (contract-first) – повторяющийся мотив всего проекта: так делают и REST, и gRPC, и события.

Этап 3.1 · REST

Этап 1. REST API contract – модуль api-contract

Что это и зачем отдельный модуль

REST – самый привычный способ: клиент шлёт HTTP-запрос (GET, POST) на адрес вроде /api/players и получает JSON. Контракт – это формальное описание: какие есть адреса, что они принимают и что возвращают. В нашем проекте контракт вынесен в отдельный модуль api-contract, который не запускается, а только подключается к matchmaking.

📐 DTO

«Формы» данных: что приходит в запросе и что уходит в ответе (records).

🔌 Интерфейсы-эндпоинты

Java-интерфейсы с аннотациями Spring и Swagger – описание URL без реализации.

⚠️ Исключения

Доменные ошибки (не найдено, нельзя войти и т.п.), общие для контракта.

🎯
Зачем выносить контракт в отдельный jar

Контракт можно отдать другой команде, другому сервису или фронтенду без исходников бизнес-логики. Реализация в matchmaking просто implements PlayerApi – и компилятор гарантирует, что она не разойдётся с контрактом. Это и есть «сначала контракт».

📁 api-contract/src/main/java/edu/rutmiit/demo/matchmakingapicontract/
Что делает
Хранит DTO, интерфейсы-контроллеры, исключения и Swagger-аннотации. GraphQL-схема лежит в src/main/resources/graphql/schema.graphqls.
Почему так
Контракт отделён от реализации → его легко переиспользовать и нельзя «случайно» нарушить.
Запускается?
Нет. Это библиотека (.jar), зависимость для matchmaking.
Как проверить
После запуска matchmaking открыть Swagger UI – он построен ровно из этих интерфейсов и DTO.

DTO – «формы» данных

DTO (Data Transfer Object) – это простой объект для передачи данных «по проводу». В проекте они сделаны как Java record (короткий неизменяемый класс). Обратите внимание на аннотации валидации (@NotBlank, @Min) – Spring проверит их автоматически.

        @NotBlank String nickname,        // обязателен, не пустой

        String region,
        String rank

) {}

Учебная деталь

Время везде – OffsetDateTime (с зоной), а не LocalDateTime. Это правило проекта: так дата корректно сериализуется в JSON и в события RabbitMQ.

matchmaking · service/MatchmakingService.java
public synchronized LobbyResponse joinLobby(JoinLobbyRequest request) {
    processExpiredLobbiesInternal();                 // (1) сначала чистим протухшие лобби

    PlayerState player = playerState(request.playerId());
    String mode   = normalize(request.mode());        // (2) приводим к верхнему регистру
    String region = normalize(request.region());
    String rank   = normalize(request.rank());
    int timeoutSeconds = request.timeoutSeconds() == null
            ? DEFAULT_TIMEOUT_SECONDS : request.timeoutSeconds();

    ensureAllowedMode(mode);                          // (3) режим из белого списка?
    ensurePlayerCanJoinQueue(player.id);              // (4) игрок ещё не ждёт и не играет?

    boolean[] created = {false};
    LobbyState lobby = lobbies.values().stream()      // (5) ищем ПОДХОДЯЩЕЕ лобби
            .filter(c -> "WAITING".equals(c.status))
            .filter(c -> c.playerIds.size() < MAX_PLAYERS_IN_LOBBY)
            .filter(c -> c.mode.equals(mode))
            .filter(c -> c.region.equals(region))
            .filter(c -> c.rank.equals(rank))
            .min(Comparator.comparing(c -> c.createdAt))  // самое старое
            .orElseGet(() -> {                        // (6) нет подходящего -> создаём новое
                created[0] = true;
                return createLobby(mode, region, rank, timeoutSeconds);
            });

    if (created[0]) lobbyEventPublisher.publishCreated(toLobbyResponse(lobby)); // событие lobby.created

    lobby.playerIds.add(player.id);                   // (7) добавляем игрока
    LobbyResponse afterJoin = toLobbyResponse(lobby);
    lobbyEventPublisher.publishPlayerJoined(afterJoin, toPlayerResponse(player)); // lobby.player-joined

    if (lobby.playerIds.size() >= MAX_PLAYERS_IN_LOBBY) {   // (8) набралось 5 -> старт сразу
        cancelLobbyStartTask(lobby.id);
        return formLobbyAndStartMatch(lobby.id);
    }

    if (lobby.playerIds.size() >= MIN_PLAYERS_TO_START) {   // (9) набралось 3 или 4 -> ждём 10 секунд
        scheduleLobbyStart(lobby.id);
    }
    return afterJoin;
}
(1)перед любой операцией распускаем лобби, у которых вышло время и не набралось 3 игроков.
(2)normalize приводит строки к единому виду (верхний регистр), чтобы eu и EU попадали в одно лобби.
(3)проверка режима по белому списку ALLOWED_MODES – иначе ошибка.
(4)ключевая защита: если игрок уже ждёт в WAITING-лобби или играет – войти повторно нельзя.
(5)лобби «свои» только если совпали все три: mode + region + rank.
(6)если подходящего нет – создаём новое и публикуем lobby.created.
(8)если игроков уже 5, стартуем матч сразу, потому что это максимум лобби.
(9)если игроков 3 или 4, запускаем ожидание 10 секунд. Новый 4-й игрок перезапустит таймер.
synchronizedметод синхронизирован – два игрока не «влезут» в лобби одновременно и не сломают счётчик.
🧩
Почему лобби группируется по mode + region + rank

Чтобы игроки попадали в матч с похожими и из своего региона. SILVER из EU в режиме BATTLE_ROYALE не встанет в одно лобби с GOLD из NA. Это три «координаты» очереди.

Почему после 3-го игрока есть ожидание 10 секунд

Как только в лобби набирается MIN_PLAYERS_TO_START = 3, матч ещё не стартует мгновенно. Сервис запускает таймер на 10 секунд, чтобы успел присоединиться 4-й игрок. Если 4-й игрок пришёл, таймер начинается заново. Если лобби набрало MAX_PLAYERS_IN_LOBBY = 5, ожидать больше не нужно: лобби сразу переходит в FORMED и создаётся матч.

Старт матча: startMatch

Лобби становится FORMED, рождается MatchState со статусом IN_PROGRESS, а событие MATCH_STARTED остаётся внутри match.events как история матча. В RabbitMQ оно отдельно не публикуется.

matchmaking · service/
private MatchState startMatch(LobbyState lobby) {
    OffsetDateTime startedAt = OffsetDateTime.now();
    lobby.status = "FORMED";                  // лобби больше не WAITING
    // startedAt сохраняет время формирования лобби

    UUID matchId = UUID.randomUUID();
    List<UUID> matchPlayerIds = List.copyOf(lobby.playerIds);
    List<MatchEventResponse> events = new ArrayList<>();

    MatchEventResponse startEvent = event("MATCH_STARTED",
            "Матч начался. Игроков в лобби: " + matchPlayerIds.size(),
            null, null, startedAt);
    events.add(startEvent);

    MatchState match = new MatchState(matchId, lobby.id, lobby.mode, lobby.region,
            lobby.rank, "IN_PROGRESS", matchPlayerIds, events, null, startedAt, null);

    matches.put(match.id(), match);
    lobby.matchId = match.id();
    return match;
}

«Симуляция» матча по таймеру: scheduleMatchSimulation

Реального игрового движка нет. Вместо него – планировщик: он раскидывает 4–8 случайных событий боя по случайным моментам внутри 60–120 секунд, а в конце вызывает finishMatch.

matchmaking · service/
private void scheduleMatchSimulation(UUID matchId) {
    ThreadLocalRandom random = ThreadLocalRandom.current();
    int durationSeconds = random.nextInt(MATCH_MIN_DURATION_SECONDS, MATCH_MAX_DURATION_SECONDS + 1); // 60..120
    int progressEvents  = random.nextInt(MATCH_MIN_PROGRESS_EVENTS,  MATCH_MAX_PROGRESS_EVENTS + 1);  // 4..8

    // раскидываем progress-события по случайным секундам матча
    for (Integer delaySeconds : randomProgressDelays(durationSeconds, progressEvents)) {
        matchExecutor.schedule(() -> addRandomMatchProgressEvent(matchId), delaySeconds, TimeUnit.SECONDS);
    }
    // в конце матча выбираем победителя
    matchExecutor.schedule(() -> finishMatch(matchId), durationSeconds, TimeUnit.SECONDS);
}

Финал матча: finishMatch

Выбирается случайный победитель, для проигравших добавляются события PLAYER_ELIMINATED, матч переходит в FINISHED, и – важнейший момент – публикуется match.finished.

matchmaking · service/
private synchronized void finishMatch(UUID matchId) {
    MatchState match = matches.get(matchId);
    if (match == null || !"IN_PROGRESS".equals(match.status())) return;

    OffsetDateTime finishedAt = OffsetDateTime.now();
    PlayerState winner = randomPlayer(match.playerIds());   // случайный победитель

    for (UUID playerId : match.playerIds()) {               // выбываем всех, кроме победителя
        if (playerId.equals(winner.id)) continue;
        MatchEventResponse elim = event("PLAYER_ELIMINATED",
                winner.nickname + " выбил игрока " + playerState(playerId).nickname,
                winner.id, playerId, finishedAt);
        match.events().add(elim);
        matchEventPublisher.publishProgress(toMatchResponse(match), elim); // match.progress
    }

    MatchEventResponse finishEvent = event("MATCH_FINISHED",
            "Матч завершен. Победитель: " + winner.nickname, winner.id, null, finishedAt);
    match.events().add(finishEvent);

    MatchState finished = new MatchState(/* ... */ "FINISHED", /* winner.id, finishedAt */);
    matches.put(matchId, finished);

    matchEventPublisher.publishFinished(toMatchResponse(finished));               // match.finished  <-- ВАЖНО
}

Этап 3.3 · GraphQL

Этап 3. GraphQL – гибкое чтение и мутации

Зачем GraphQL, если уже есть REST?

REST отдаёт «фиксированные» ответы: запросил игрока – получил все его поля, даже если нужно одно. GraphQL переворачивает это: клиент сам описывает, какие поля ему нужны, и получает ровно их – ни больше, ни меньше. Один запрос может вытащить лобби вместе с игроками и связанным матчем за один поход.

REST

Сервер решает форму ответа. Несколько эндпоинтов. Чтобы собрать связанные данные – несколько запросов.

GraphQL

Клиент решает форму ответа. Один эндпоинт /graphql. Связанные данные – в одном запросе через вложенные поля.

🧠
Главное правило проекта

GraphQL не дублирует бизнес-логику. Его резолверы (DataFetchers) просто вызывают тот же MatchmakingService, что и REST-контроллеры. Одна правда – два «окна» в неё.

Этап 3.4 · RabbitMQ

Этап 4. RabbitMQ и контракт событий

Зачем шина событий и почему асинхронно

До сих пор связь была синхронной: спросил – подождал – получил. Но «сообщить всем, что матч завершён» так делать неудобно: главный сервис не должен знать, кто и зачем слушает. Решение – шина событий. matchmaking просто публикует событие в RabbitMQ и забывает о нём. Кто подписан – тот и получит, когда сможет.

Синхронно (REST/gRPC)

«Позвонить и ждать ответа». Отправитель знает получателя и блокируется до ответа.

Асинхронно (RabbitMQ)

«Бросить письмо в почту». Отправитель не знает получателей и не ждёт. Слабая связанность, устойчивость к сбоям.

📌
Publisher не знает Consumer

В этом вся суть. matchmaking не импортирует match-log-service и не зовёт его. Он шлёт match.finished в exchange – а уж кто подписался (лог, уведомления, enrichment), его не касается.

}

Этап 5. match-log-service – журнал событий

Почему «audit-service» стал «match-log-service»

В эталоне преподавателя этот сервис назывался audit-service (аудит действий с книгами). ✅ Что он делает

  • хранит их в in-memory журнале по порядку;
  • отдаёт журнал по REST;
  • отбрасывает дубликаты по eventId.

🚫 Чего он НЕ делает

  • ничего не считает и не меняет;
  • не вызывает другие сервисы;
  • не хранит данные в БД;
  • не влияет на ход матча.

Этап 3.6 · gRPC

Этап 6. gRPC – contract-first обогащение рейтинга

Зачем gRPC и почему есть и клиент, и сервер

RabbitMQ хорош для «оповестить всех». Но иногда нужен точный синхронный вызов «посчитай и верни ответ» между двумя сервисами. Для этого – gRPC: быстрый бинарный протокол поверх HTTP/2, описанный контрактом .proto. У нас два участника:

🖥️ grpc-match-analytics-server

gRPC-сервер на порту 9090. Умеет одно: посчитать изменения рейтинга после матча. Не слушает RabbitMQ.

🔌 grpc-match-enrichment-client

gRPC-клиент и одновременно RabbitMQ-мост. Ловит match.finished, зовёт сервер, публикует match.enriched.

public class MatchAnalyticsServiceImpl extends MatchRankEnrichmentGrpc.MatchRankEnrichmentImplBase {
    @Override
    public void calculateRankChanges(CalculateRankChangesRequest request,
                                     StreamObserver<RankChangesResponse> responseObserver) {
        RankChangesResponse.Builder response = RankChangesResponse.newBuilder()
                .setMatchId(request.getMatchId())
                .setWinnerNickname(request.getWinnerNickname());

        for (int i = 0; i < request.getPlayersCount(); i++) {
            MatchPlayer p = request.getPlayers(i);
            int delta = p.getWinner() ? 25 : -10;             // победитель +25, остальные -10
            int newRating = Math.max(0, p.getRating() + delta); // не уходим в минус
            String newRank = rankByRating(newRating);           // те же пороги, что в matchmaking

            response.addChanges(PlayerRankChange.newBuilder()
                    .setPlayerId(p.getPlayerId()).setNickname(p.getNickname())
                    .setOldRating(p.getRating()).setDelta(delta).setNewRating(newRating)
                    .setOldRank(p.getRank()).setNewRank(newRank).build());
        }
        responseObserver.onNext(response.build());   // отдать ответ
        responseObserver.onCompleted();
    }
}

После расчёта рейтинги игроков в matchmaking обновляются. Сервис слушает match.enriched, читает изменения рейтинга и меняет игроков в своей players HashMap. Это сделано максимально просто: без базы данных, без отдельного сервиса записи и без усложнения архитектуры.

Что считает сервер – наглядно

Пример: матч из 3 игроков (старт по 1210). Победитель: 1210 -> 1235 (+25). Двое проигравших: 1210 -> 1200 (-10).
Мини-квиз gRPC
После обогащения вы открываете /api/players и видите старый рейтинг. Почему?
Верно. Enrichment-client публикует match.enriched, а matchmaking слушает это событие и применяет изменения к игрокам в памяти.
Этап 3.7 · WebSocket

Этап 7. notification-service – живые WebSocket-уведомления

Зачем WebSocket и чем он лучше «опроса» (polling)

Как показать в браузере уведомление «матч завершён» сразу? С обычным REST браузер пришлось бы каждую секунду спрашивать «ну что, есть новости?» (polling) – это расточительно. WebSocket – это постоянное двустороннее соединение: браузер один раз подключился, и теперь сервер сам шлёт сообщения, как только они появятся.

REST polling

Браузер дёргает сервер «есть новое?» снова и снова. Задержки, лишний трафик.

WebSocket push

Соединение держится открытым. Сервер пушит сообщение мгновенно. Идеально для уведомлений.

Паттерн «событие RabbitMQ → push в браузер»

notification-service – это мост между шиной событий и браузером: он слушает RabbitMQ (как match-log) и рассылает полученное всем подключённым WebSocket-клиентам.

private final Set<WebSocketSession> sessions = ConcurrentHashMap.newKeySet();

public void broadcast(String json) {
    TextMessage message = new TextMessage(json);
    for (WebSocketSession s : sessions) {
        if (!s.isOpen()) { sessions.remove(s); continue; }
        try { s.sendMessage(message); }                 // рассылка КАЖДОМУ клиенту
        catch (IOException e) { sessions.remove(s); }
    }
}

const WS_URL = `ws://${window.location.host}/ws/notifications`;
function connect() {
  ws = new WebSocket(WS_URL);
  ws.onopen    = () => setStatus('connected', 'Connected');
  ws.onmessage = (e) => {
    const data = JSON.parse(e.data);
    if (data.type === 'CONNECTED' || data.type === 'PONG') return; // служебные
    addNotification(data);                                          // показать карточку
  };
  ws.onclose   = () => { setStatus('disconnected'); scheduleReconnect(); };
}
function addNotification(data) {
  notificationCount++;
  const card = document.createElement('div');
  card.className = `card ${data.level || 'info'}`;     // цвет по уровню
  card.innerHTML = `<div class="title">${data.title}</div>
                    <div class="description">${data.description}</div>
                    <div class="meta">event: ${data.eventType} · id: ${data.eventId}</div>`;
  list.prepend(card);   // новые сверху, в памяти страницы
}
function scheduleReconnect() {
  const delay = Math.min(1000 * Math.pow(2, reconnectAttempt), 30000); // 1s,2s,4s... до 30s
  reconnectAttempt++;
  setTimeout(connect, delay);
}

Флешкарты: типичные вопросы преподавателя

Нажмите на карточку, чтобы увидеть ответ.

Вопрос
Почему контрактов три и они не запускаются?
нажми →
Контракты (api-contract, events-contract, grpc-contract) – это общие «словари» типов. Они собираются в jar и подключаются к сервисам, чтобы все говорили одинаково. У них нет main(), поэтому запустить нельзя.
Вопрос
Что значит match.finished?
нажми →
Несмотря на название, это «матч завершён, готов финальный snapshot» – событие публикуется в конце finishMatch. Оно запускает gRPC-обогащение. Начало матча – это lobby.formed.
Вопрос
Почему рейтинг меняется не сразу в момент финиша?
нажми →
Сначала публикуется match.finished, затем enrichment-client вызывает gRPC и публикует match.enriched. Только после обработки match.enriched сервис matchmaking обновляет игроков в памяти.
Вопрос
Зачем и REST, и GraphQL?
нажми →
Это два «окна» в один сервис. REST – фиксированные ответы и Swagger. GraphQL – клиент сам выбирает поля и тянет связанные данные одним запросом. Оба вызывают один MatchmakingService.
Вопрос
Чем gRPC отличается от RabbitMQ?
нажми →
gRPC – синхронный вызов «запрос→ответ» между двумя сервисами (быстрый, строгий контракт .proto). RabbitMQ – асинхронная рассылка «одному-многим», отправитель не ждёт и не знает получателей.
Вопрос
Почему матч не стартует сразу на 3 игроках?
нажми →
MIN_PLAYERS_TO_START = 3, но после 3-го игрока есть ожидание 10 секунд. 4-й игрок может успеть войти и перезапустить таймер. MAX_PLAYERS_IN_LOBBY = 5, поэтому 5-й запускает матч сразу.
Вопрос
Чем eventId отличается от matchId?
нажми →
eventId – id сообщения в RabbitMQ (для трассировки и дедупликации). matchId – id доменной сущности «матч». Разные миры: одно сообщение про один матч имеет свой eventId.
Вопрос
Почему данные в памяти?
нажми →
Цель курса – показать связь сервисов, а не БД. Игроки/лобби/матчи в LinkedHashMap; после рестарта сбрасываются. Это сознательное упрощение, заявленное в правилах проекта.

Минимальные классы по этапам

  1. Каркас. Корневой pom.xml (packaging=pom, список <modules>). Модуль api-contract: записи PlayerRequest/Response, JoinLobbyRequest, LobbyResponse, MatchResponse; интерфейсы PlayerApi/LobbyApi/MatchApi. Модуль matchmaking с DemoApplication.
  2. REST. MatchmakingService с Map-хранилищами, createPlayer, rankByRating, joinLobby, startMatch, finishMatch, планировщик. Контроллеры implements *Api + HATEOAS. Подключить Swagger. Проверка: Swagger создаёт игрока и собирает матч.
  3. GraphQL. Добавить schema.graphqls, зависимость DGS, @DgsComponent-резолверы, что зовут MatchmakingService; вычисляемое averageRating; scalar DateTime; instrumentation. Проверка: GraphiQL.
  4. RabbitMQ. Модуль matchmaking-events-contract: EventEnvelope, EventMetadata, RoutingKeys, *Event. В matchmaking: RabbitMQConfig (topic exchange) + три publisher’а. Проверка: сообщения в RabbitMQ UI.
  5. match-log-service. Новый сервис: @RabbitListener на очередь #, in-memory storage с дедупликацией, REST /api/match-log. Проверка: история событий по адресу.
  6. gRPC. Модуль matchmaking-grpc-contract с .proto + protobuf-плагин. grpc-match-analytics-server (реализация + SmartLifecycle на 9090). grpc-match-enrichment-client (listener match.finished → blocking stub → publish match.enriched). Проверка: gRPC-логи.
  7. WebSocket. notification-service: WebSocketConfig, handler с broadcast, listener (#) → NotificationPayload, статичный index.html. Проверка: живые карточки в браузере.

«Сначала это, потом то»

На каждом этапе добейтесь, чтобы работало и проверяемо, прежде чем идти дальше. Сломалось на gRPC? REST и RabbitMQ уже работают – значит, проблема локальна. Это и есть инкрементальная сборка.

Чего НЕ добавлять (правила проекта)

Никакой базы данных, JWT/security, API Gateway, сложной игровой симуляции. Это учебный проект – лишняя инфраструктура только мешает объяснять архитектуру.

Раздел 8 · Словарь

Глоссарий простыми словами

ТерминПростыми словами
Service (сервис)Отдельная запускаемая программа с одной зоной ответственности и своим портом.
Contract (контракт)Описание «как с нами разговаривать» (типы и операции), вынесенное отдельно от реализации.
DTOData Transfer Object – простой объект «формы данных» для передачи по сети (у нас – record).
ControllerКласс, принимающий HTTP-запрос и отдающий ответ. Логики в нём нет – он зовёт сервис.
Service layerСлой бизнес-логики (MatchmakingService): здесь живут правила (лобби, матчи, ранги).
In-memory storageДанные хранятся в обычных коллекциях в оперативной памяти. Без БД; сбрасываются при рестарте.
HATEOASКогда ответ содержит не только данные, но и ссылки на связанные действия (_links).
GraphQL Query / MutationQuery – чтение, Mutation – изменение. Клиент сам указывает, какие поля вернуть.
DataFetcherМетод-резолвер GraphQL (@DgsQuery/@DgsMutation/@DgsData), который добывает данные для поля.
Exchange«Сортировочный узел» RabbitMQ: получает сообщение и раскидывает его по очередям. У нас тип topic.
Queue (очередь)Буфер, где сообщения ждут, пока consumer их заберёт.
BindingПравило «эта очередь получает сообщения с такими routing key» (например # или match.finished).
Routing key«Тема» сообщения (match.finished), по которой exchange решает, в какие очереди его класть.
DLQ / DLXDead-Letter Queue/Exchange – куда уходят сообщения, которые не удалось обработать. Чтобы не терять и не зацикливать.
Event envelope«Конверт» EventEnvelope: метаданные (EventMetadata) + полезная нагрузка (payload).
eventIdУникальный id сообщения в RabbitMQ (для трассировки/дедупликации). Не путать с matchId.
gRPC server / clientСервер – кто выполняет удалённый метод; клиент – кто его вызывает по сети.
stubСгенерированный объект-«пульт» на стороне клиента: вызываешь его метод – он шлёт gRPC-запрос на сервер.
proto / protobufЯзык описания gRPC-контракта (.proto) и бинарный формат данных. Из него генерируется Java.
WebSocketПостоянное двустороннее соединение браузер↔сервер. Сервер сам шлёт сообщения (push).
NotificationЖивое уведомление в браузере. UI хранит последние 50 карточек в localStorage, а история событий – в match-log.
Maven aggregatorКорневой pom.xml (packaging=pom) со списком модулей – он сам ничего не делает, но собирает всё вместе.
ReactorМеханизм Maven, который строит порядок сборки модулей по их зависимостям.
-pl / -am-pl – собрать/запустить указанный модуль; -am – заодно собрать те, от которых он зависит.

«Один сценарий матчмейкинга – пять способов связи: REST/GraphQL на входе в matchmaking, RabbitMQ для асинхронной рассылки событий, gRPC для синхронного расчёта рейтинга, WebSocket для живых уведомлений. Главный сервис публикует доменные события и дополнительно слушает match.enriched, чтобы обновить рейтинг игроков в памяти. Данные – in-memory, потому что цель – архитектура, а не БД. match.finished = "матч завершён, готов финал".»

Три вещи, которые легко забыть на защите

1) match.finished – это конец матча, не начало. 2) matchmaking обновляет рейтинг после события match.enriched. 3) Notification UI хранит последние 50 карточек в localStorage, а полная история остаётся в match-log-service.
